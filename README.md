# 🧱 BRICKStack Studio — Multi-Agent AI Coding Platform

A multi-agent AI coding platform with chat UI, persistent terminal sandbox, and 5-agent orchestration.

## Features

- 💬 **Chat UI** — Rich text bubbles, editable code blocks, re-run capability
- 🤖 **5-Agent Orchestrator** — Planner → Coder → Terminal → Reviewer → Writer
- 🐳 **Docker Sandbox** — Isolated code execution with PTY terminal
- 📁 **File Tree Workspace** — Browse files generated by agents
- 💾 **Persistent Storage** — PostgreSQL + Redis (with in-memory fallback for dev)
- 🔐 **Session Management** — JWT-based auth + rate limiting

## Architecture

```
brickstack/
├── frontend/          # HTML/JS/CSS chat UI (single page app)
│   ├── index.html
│   ├── app.js
│   └── style.css
├── backend/           # FastAPI + WebSocket gateway
│   ├── main.py        # API server + WS handler
│   └── auth.py        # Sessions + rate limiting
├── orchestrator/      # LangGraph multi-agent brain
│   ├── graph.py       # Agent flow control
│   ├── agents.py      # 5 agent definitions
│   └── schemas.py     # State & message types
├── sandbox/           # Docker execution environment
│   ├── Dockerfile
│   └── runner.py      # PTY terminal manager (mock fallback in dev)
├── storage/           # Persistence layer
│   ├── db.py          # DB clients (Postgres/Redis + in-memory fallback)
│   └── models.py      # Chat & session models
├── docker-compose.yml # Full stack with Postgres, Redis, Sandbox
├── Dockerfile         # Production image
└── start.sh           # Quick local start script
```

## Data Flow

```
User types → WebSocket → FastAPI
  → LangGraph orchestrator
    → PlannerAgent (breaks task into steps)
    → CoderAgent (writes code)
    → TerminalAgent (runs in sandbox, streams output)
    → ReviewerAgent (checks quality)
    → WriterAgent (formats final answer)
  → Streams back over WebSocket
  → Frontend renders: locked rich text + editable code blocks
```

## Quick Start

### Option 1: Local Development (No Docker)

Requirements: Python 3.10+, Node.js (optional, for frontend dev)

```bash
# Clone
git clone https://github.com/kerollosmakary-ai/BRICKStack.git
cd BRICKStack

# Install dependencies
pip install -r backend/requirements.txt

# Run (uses in-memory DB, no Docker needed)
./start.sh
```

Open http://localhost:8000 — the frontend and backend are served together.

### Option 2: Docker Compose (Full Stack)

```bash
# Copy env file
cp .env.example .env
# Edit .env — add your LLM API key

# Start everything
docker-compose up --build
```

Services:
- **App**: http://localhost:8000
- **Postgres**: localhost:5432
- **Redis**: localhost:6379
- **Sandbox**: Docker-in-Docker isolated execution

## Environment Variables

Copy `.env.example` to `.env` and configure:

| Variable | Default | Description |
|----------|---------|-------------|
| `API_HOST` | `0.0.0.0` | API bind address |
| `API_PORT` | `8000` | API port |
| `LLM_API_KEY` | `sk-your-key` | OpenAI/DeepSeek API key |
| `LLM_BASE_URL` | `https://api.deepseek.com/v1` | LLM provider endpoint |
| `LLM_MODEL` | `deepseek-chat` | Model name |
| `DATABASE_URL` | `postgresql+asyncpg://...` | Postgres connection |
| `REDIS_URL` | `redis://localhost:6379/0` | Redis connection |
| `JWT_SECRET` | `change-me` | JWT signing secret |
| `SANDBOX_IMAGE` | `brickstack-sandbox` | Docker image for sandbox |

## WebSocket Protocol

**Send:**
```json
{"type": "user_message", "content": "write a python script to count files"}
```

**Receive events:**
```json
{"type": "thought", "agent": "Planner", "content": "Analyzing..."}
{"type": "code", "agent": "Coder", "language": "python", "code": "..."}
{"type": "terminal", "agent": "Terminal", "output": "..."}
{"type": "assistant", "agent": "Writer", "content": "...", "done": true}
```

**Edit & re-run:**
```json
{"type": "edit_code", "block_id": "abc123", "new_source": "print('hello')", "auto_run": true}
```

## Development Notes

### In-Memory Mode (Default for Local Dev)
When `DATABASE_URL` is set to `sqlite:///:memory:` or unavailable, the app falls back to:
- In-memory message storage
- Session state in Python dicts
- Mock sandbox runner (no Docker required)

This is perfect for frontend development and testing the agent pipeline.

### Adding Real LLM Integration

The agents in `orchestrator/agents.py` currently use **mock implementations**. To connect to a real LLM:

1. Set `LLM_API_KEY` and `LLM_BASE_URL` in `.env`
2. Replace the mock logic in each agent with `openai.AsyncOpenAI` or `langchain` calls
3. The graph in `orchestrator/graph.py` already supports async streaming

Example pattern for `CoderAgent`:
```python
from openai import AsyncOpenAI
client = AsyncOpenAI(api_key=os.getenv("LLM_API_KEY"), base_url=os.getenv("LLM_BASE_URL"))

async def run(self, state):
    response = await client.chat.completions.create(
        model=os.getenv("LLM_MODEL"),
        messages=[{"role": "user", "content": state["messages"][-1]["content"]}]
    )
    return {"code": response.choices[0].message.content}
```

### Sandbox Security

The sandbox uses Docker containers with:
- Memory limits (`SANDBOX_MEMORY`)
- CPU limits (`SANDBOX_CPU`)
- Timeout (`SANDBOX_TIMEOUT`)
- No network access (in production)

For production, consider:
- gVisor or Firecracker for stronger isolation
- Network namespaces + iptables
- File system restrictions (read-only root, tmpfs for /tmp)
